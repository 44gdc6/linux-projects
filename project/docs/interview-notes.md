# 简历与面试讲稿

## 简历写法

项目名称建议：

**基于嵌入式 Linux 的多传感器监测与告警系统**

可直接放简历的项目描述：

1. 独立完成 LM75、MQ3、ADXL345、DHT11 和蜂鸣器等外设的 Linux 驱动开发，覆盖 I2C、SPI、platform/GPIO、IIO consumer 等多种接入方式，并统一通过 misc device 暴露 `/dev` 接口。
2. 基于 C 和 `pthread` 设计用户态多线程架构，实现传感器采集、天气缓存、SQLite 存储、MQTT 上报和 LVGL 界面刷新等模块协同。
3. 打通从驱动原始数据到用户态阈值判断、蜂鸣器告警联动、本地持久化和云端上报的完整链路。
4. 完成驱动加载、节点验证、最小读写测试和整机联调，能够定位设备绑定、读取失败和异常恢复问题。

## 两分钟项目介绍

推荐叙事顺序：

1. 这是一个嵌入式 Linux 多传感器项目，核心目标是把多类外设统一接入用户态系统。
2. 我自己实现了 LM75、MQ3、ADXL345、DHT11 和蜂鸣器驱动，分别覆盖了 I2C、SPI、GPIO/platform 和 IIO consumer 这些典型场景。
3. 驱动层统一通过 misc device 提供 `/dev` 接口，用户态再按设备访问、采集调度、存储、上报和显示分层。
4. 最终系统可以完成采集、告警、蜂鸣器联动、SQLite 落库和 MQTT 上报。

## 高频追问

### 为什么选 misc device？

- 节点创建简单
- 用户态调试直接
- 对项目展示来说，能把重点放在驱动接入和数据通路，而不是设备号管理

### 各驱动分别绑定了什么？

- LM75：I2C driver
- ADXL345：SPI driver
- DHT11：platform + GPIO
- MQ3：platform + IIO consumer
- 蜂鸣器：GPIO 输出控制 + misc device

### 驱动和用户态职责怎么划分？

- 驱动：绑定硬件、读取或设置原始状态、暴露 `/dev` 接口
- 用户态：解析数据、做阈值判断、触发告警、存储、上报和显示

### 你修过什么问题？

这轮优化里很适合讲的一个点：

- 之前有些驱动 `read()` 成功了但返回 `0`，这对字符设备语义和用户态判断都不友好
- 我把它们统一成返回实际字节数，并补上长度检查、标准错误码和更清晰的日志

### 你怎么验证驱动是好的？

- `insmod` / `dmesg`
- `/dev` 节点检查
- 最小 userspace 读写验证
- 完整应用联调

### 设备读取失败时怎么办？

- 用户态会在可能的地方重试或做退化处理
- 例如温度采集当前已经保留 mock fallback，保证主流程不会因为单个设备异常而完全中断

## 面试时尽量少这样讲

- 不要把项目重点讲成“MQTT + SQLite + UI”
- 对驱动岗来说，更重要的是：
  - 设备怎么绑定
  - 原始数据怎么到用户态
  - 接口怎么设计
  - 异常怎么处理
