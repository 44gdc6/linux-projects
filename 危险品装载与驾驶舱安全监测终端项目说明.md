# 基于 i.MX6ULL 的危险品装载与驾驶舱安全监测终端项目说明

## 一、项目背景与定位

危险品运输车辆在实际运行过程中，风险主要集中在两个区域：

- 装载区：危险品罐体、阀门、舱门、轮边等位置可能出现气体泄漏、温度异常、局部烟雾或火源、非法开舱等问题
- 驾驶舱：驾驶员可能出现酒精驾驶、驾驶舱烟雾异常、温湿度异常、设备运行状态异常等问题

因此，本项目不再单纯把“危险车辆检测”理解为车辆本体状态监测，而是将系统划分为 `危险品装载区监测` 和 `驾驶舱安全监测` 两条主线，构建一套基于 `NXP i.MX6ULL` 的 ARM + Linux 综合监测终端。

## 二、系统总体架构

系统整体采用六层结构：

```text
感知层 -> Linux驱动与接口层 -> 设备抽象层 -> 业务逻辑层 -> 本地交互层 -> 平台通信层
```

### 1. 感知层

感知层分为两部分：

- 装载区监测：面向危险品装载本体和车辆关键部位
- 驾驶舱监测：面向驾驶员和驾驶环境安全

### 2. Linux 驱动与接口层

接口层负责完成所有传感器和外设的 Linux 侧接入，重点接口包括：

- `GPIO`
- `I2C`
- `UART`
- `SPI`
- `ADC`
- `CAN`
- `Ethernet`

### 3. 设备抽象层

负责对传感器和节点进行统一封装，包括：

- 传感器初始化
- 数据读取接口统一
- 节点状态管理
- 在线状态检测
- 数据缓存管理

### 4. 业务逻辑层

业务逻辑层负责：

- 采集调度
- 装载区风险判断
- 驾驶舱风险判断
- 分级告警
- 本地联动和远程上报

### 5. 本地交互层

主要包括：

- LCD 显示实时状态
- 蜂鸣器与 LED 声光报警
- 继电器控制外部告警设备
- 本地日志记录

### 6. 平台通信层

通过以太网将车辆状态同步到平台，使用 `MQTT` 完成：

- 周期状态上传
- 异常告警上传
- 心跳保活
- 历史数据追溯

## 三、硬件组成与传感器选型

### 1. 主控平台

- 主控芯片：`NXP i.MX6ULL`
- 核心架构：`ARM Cortex-A7`
- 操作系统：`Embedded Linux`
- 典型接口：`GPIO、I2C、UART、SPI、ADC、CAN、Ethernet、USB`

### 2. 装载区监测传感器

| 监测对象 | 器件型号 | 接口 | 作用 |
| --- | --- | --- | --- |
| 可燃气体检测 | MQ-5 | ADC | 检测罐体周边可燃气体浓度 |
| 有害气体辅助检测 | MQ-135 | ADC | 辅助检测有害气体和空气质量变化 |
| 轮边温度检测 | DS18B20 | GPIO / 单总线 | 检测轮边、制动部位温度 |
| 高温点检测 | MAX6675 + K型热电偶 | SPI | 检测制动鼓、轴承座等高温部位 |
| 舱门状态检测 | 霍尔门磁开关 | GPIO | 检测舱门是否异常开启 |
| 姿态振动检测 | MPU6050 | I2C | 检测车辆急刹、碰撞、侧翻趋势 |
| 烟雾检测 | MQ-2烟雾传感器模块 | ADC / GPIO | 检测局部烟雾异常 |
| 火焰检测 | 火焰传感器模块 | GPIO / ADC | 检测局部火源 |

### 3. 驾驶舱监测传感器

| 监测对象 | 器件型号 | 接口 | 作用 |
| --- | --- | --- | --- |
| 酒精检测 | MQ-3 | ADC | 检测驾驶员酒精挥发气体浓度 |
| 温湿度检测 | AHT20 | I2C | 检测驾驶舱环境温湿度 |
| 烟雾检测 | 烟雾传感器模块 | ADC / GPIO | 检测驾驶舱烟雾异常 |
| 心率/血氧检测 | MAX30100 | I2C | 辅助检测驾驶员心率与血氧状态 |
| 气压/温度检测 | BMP280 | SPI | 辅助监测驾驶舱气压和温度变化 |
| GPS 定位 | NEO-6M | UART | 获取位置、速度、时间信息 |

### 4. 通信与执行模块

| 模块类型 | 器件型号 | 接口 | 作用 |
| --- | --- | --- | --- |
| CAN 扩展模块 | MCP2515 + TJA1050 | SPI / CAN | 构建分布式采集节点 |
| 本地告警模块 | 蜂鸣器 + LED + 继电器 | GPIO / PWM | 本地联动告警 |
| 显示模块 | 4.3 寸 LCD | RGB / SPI | 实时显示装载区与驾驶舱状态 |
| 网络通信模块 | 开发板以太网口 | Ethernet | 远程联网与 MQTT 上传 |

### 5. 选型思路

本项目的器件选型遵循以下原则：

- 优先覆盖 ARM + Linux 岗位常见接口
- 同时兼顾装载区与驾驶舱两类实际业务需求
- 选用原型阶段常见器件，便于落地与调试
- 保留后续替换工业级传感器的扩展空间
- 在 `SPI` 接口上不仅接入 `MCP2515`，还明确接入 `MAX6675` 和 `BMP280`，保证 `SPI` 协议在项目中有实际传感器应用

## 四、Linux 驱动与接口设计

本项目核心不是单纯列举器件，而是要围绕 `Linux 驱动与接口设计` 建立完整实现逻辑。

### 1. GPIO 接口设计

用于以下器件：

- 霍尔门磁开关
- 蜂鸣器
- LED
- 继电器
- 火焰传感器数字信号
- DS18B20 单总线访问

重点体现：

- 输入输出方向控制
- 电平读写
- 告警器件联动控制
- 门磁/火焰状态检测

### 2. I2C 接口设计

用于以下器件：

- AHT20
- MPU6050
- MAX30100

重点体现：

- I2C 设备树节点配置
- 从设备地址访问
- 寄存器读写
- 原始数据解析与封装
- 心率与血氧数据读取

### 3. UART 接口设计

用于以下器件：

- NEO-6M GPS

重点体现：

- 串口初始化与波特率配置
- NMEA 语句接收与解析
- 时间、经纬度、速度字段提取

### 4. SPI 接口设计

用于以下器件：

- MCP2515
- MAX6675 + K型热电偶
- BMP280（SPI 模式）

重点体现：

- SPI 控制器初始化
- MCP2515 配置与模式切换
- MAX6675 温度数据读取
- BMP280 SPI 寄存器读写
- CAN 报文发送与接收
- 节点 ID 管理
- 高温点与环境气压数据采集

### 5. ADC 接口设计

用于以下器件：

- MQ-2
- MQ-3
- MQ-135
- 烟雾传感器模拟量

重点体现：

- 模拟量采样
- 电压读取
- 浓度阈值换算
- 酒精、可燃气体、有害气体状态分类

### 6. CAN 接口设计

系统通过 `MCP2515 + TJA1050` 构建 CAN 总线，主控负责：

- 接收装载区各采集节点报文
- 按消息 ID 区分节点功能
- 将 CAN 节点数据纳入统一规则引擎
- 向告警执行节点下发控制命令

### 7. Ethernet 接口设计

通过开发板自带网口完成联网，主要实现：

- 网络初始化
- IP 配置和连通性检测
- MQTT 客户端连接
- 状态与告警上传
- 异常断开重连

### 驱动与接口流程图

```text
本地传感器 / CAN 节点
        │
        ▼
GPIO / I2C / UART / SPI / ADC / CAN / Ethernet
        │
        ▼
Linux 驱动接口访问
        │
        ▼
设备抽象层统一封装
        │
        ▼
应用线程读取数据
        │
        ▼
风险判断 / 本地联动 / MQTT 上传
```

## 五、CAN 分布式节点设计

为了体现真实车载监测系统的结构，本项目将装载区高相关的部分功能设计为 CAN 分布式节点，驾驶舱部分则主要由主控本地直连采集。

### 1. 轮边温度节点

- 传感器：`DS18B20`
- 采集内容：轮边或制动位置温度
- 节点作用：判断轮边过热、制动异常、轴承发热
- 示例消息 ID：`0x101`

### 1-1. 高温点检测节点

- 传感器：`MAX6675 + K型热电偶`
- 采集内容：制动鼓、轴承座等高温部位温度
- 节点作用：作为 `SPI` 传感器应用示例，用于检测高温危险点
- 示例消息 ID：`0x106`

### 2. 罐体气体检测节点

- 传感器：`MQ-2`
- 采集内容：罐体周边可燃气体浓度
- 节点作用：判断危险品泄漏风险
- 示例消息 ID：`0x102`

### 3. 舱门状态节点

- 传感器：`霍尔门磁开关`
- 采集内容：舱门开关状态
- 节点作用：识别非法开舱
- 示例消息 ID：`0x103`

### 4. 姿态振动节点

- 传感器：`MPU6050`
- 采集内容：三轴加速度、姿态、倾角
- 节点作用：判断碰撞、急刹、侧翻趋势
- 示例消息 ID：`0x104`

### 5. 烟雾/火焰检测节点

- 传感器：`烟雾传感器模块`、`火焰传感器模块`
- 采集内容：烟雾值、火焰状态
- 节点作用：识别局部燃烧或高温异常
- 示例消息 ID：`0x105`

### 6. 告警执行节点

- 执行器：`蜂鸣器`、`LED`、`继电器`
- 功能：接收主控指令并执行本地告警动作
- 示例控制 ID：`0x301`

## 六、应用业务闭环

本项目的核心业务闭环不是单一区域监测，而是“装载区 + 驾驶舱”双域联动。

### 1. 装载区业务闭环

```text
装载区传感器采集
-> Linux 接口读取 / CAN 节点接收
-> 气体、温度、姿态、舱门状态分析
-> 泄漏/过热/非法开舱/碰撞风险判断
-> 本地告警 + 平台告警
```

装载区主要处理：

- MQ-2 检测危险品泄漏
- DS18B20 检测轮边或关键部位温度
- 霍尔门磁检测舱门状态
- MPU6050 检测车辆姿态和振动
- 烟雾/火焰模块检测局部异常燃烧

### 2. 驾驶舱业务闭环

```text
驾驶舱传感器采集
-> 酒精/烟雾/温湿度/心率血氧/GPS 数据解析
-> 驾驶员和驾驶环境状态判断
-> 驾驶舱异常告警
-> 本地提示 + 平台记录
```

驾驶舱主要处理：

- `MQ-3` 检测驾驶员酒精挥发气体浓度
- 烟雾传感器检测驾驶舱异常烟雾
- `AHT20` 检测驾驶舱温湿度
- `MAX30100` 辅助检测驾驶员心率和血氧状态
- `NEO-6M` 提供位置和时间信息

## 八、项目着重完成内容

考虑到本项目涉及装载区监测、驾驶舱监测、Linux 驱动接口、CAN 分布式节点和网络通信等多个部分，最终实施时不需要把所有设想一次性全部做完。为了保证项目能够按期落地，最终方案中应优先聚焦“核心功能闭环”，即先完成最能体现 `ARM + Linux 驱动项目` 特点、同时最容易形成演示效果的内容。

### 1. 必须完成的核心内容

以下部分建议作为最终必须完成的核心内容：

- 完成 `i.MX6ULL` 平台下 Linux 系统运行与基础环境搭建
- 完成常见接口的基本调通，至少覆盖 `GPIO、I2C、UART、SPI、ADC、Ethernet`
- 完成装载区核心监测功能：
  - `MQ-2` 可燃气体检测
  - `DS18B20` 温度检测
  - `霍尔门磁` 舱门状态检测
  - `MPU6050` 姿态振动检测
- 完成驾驶舱核心监测功能：
  - `MQ-3` 酒精检测
  - `AHT20` 温湿度检测
  - 驾驶舱烟雾检测
- 完成本地告警联动：
  - 蜂鸣器
  - LED
  - LCD 实时状态显示
- 完成以太网通信与 MQTT 数据上报
- 完成基本日志记录与异常状态上报

### 2. 建议完成的提升内容

如果时间和进度允许，建议继续完成以下内容，用于提高项目完整性：

- 完成 `MCP2515 + TJA1050` 的 CAN 通信接入
- 至少实现 1 到 2 个 CAN 分布式节点，例如：
  - 轮边温度节点
  - 罐体气体检测节点
- 完成节点报文解析和节点在线状态监测
- 完成本地日志文件保存与简单历史查询
- 对采集数据增加阈值配置或分级告警逻辑

这些内容能够进一步突出项目在：

- 总线通信
- 分布式采集
- 车载系统结构设计

方面的深度。

### 3. 可作为扩展项的内容

以下内容可以保留在方案中，但不建议作为当前阶段的硬性完成目标：

- `MAX30100` 驾驶员心率/血氧监测
- `MAX6675 + K 型热电偶` 高温点检测
- `BMP280` 气压/温度辅助检测
- 更复杂的多节点 CAN 网络
- 平台侧完整 Web 可视化界面
- 更多工业级传感器替换方案

这些功能适合作为后续扩展方向，用于说明项目具备可升级能力，但不应影响当前核心方案落地。
