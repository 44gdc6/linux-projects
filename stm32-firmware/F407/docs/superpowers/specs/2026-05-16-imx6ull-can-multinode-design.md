# i.MX6ULL CAN 多节点挂载设计

## 目标
在同一条 CAN 总线上同时接入：

- `i.MX6ULL` 主站
- `STM32F103 + TJA1050` 节点
- `STM32F407 + TJA1050` 节点

第一版目标不是做复杂业务，而是先做出稳定、可调试、可扩展的多节点最小闭环：

1. `i.MX6ULL` 可以按节点 ID 定向发命令。
2. F103/F407 都能定时发心跳。
3. F103/F407 都能按命令回响应帧。
4. 两块 STM32 都保留串口日志，便于排错。

## 当前基线

### F103（`d:\CubemxProject\UART`）
- 已有可工作的 CAN 调试代码。
- 当前实现已经能在 `500 kbps` 下和 `i.MX6ULL` 通信。
- 启动日志、接收日志、发送异常日志都已经具备。
- 现有逻辑更像单节点演示：周期发 `0x555`，还没有统一的“节点协议”层。

### F407（`d:\CubemxProject\F407`）
- 目前只有 CubeMX 生成的初始化骨架。
- `main.c` 只初始化 GPIO、CAN1、UART5，没有运行逻辑。
- `can.c` 的时序参数和总线联调目标不匹配，也没有过滤器、启动、中断通知和应用层协议。
- `stm32f4xx_it.c` 没有 CAN 中断入口。

## 本轮选定方案
按用户选定的“方案 3”执行，这里将其具体化为：

> **统一节点协议模板，`i.MX6ULL` 做主站轮询，F103/F407 共用一套帧结构，仅通过节点 ID 区分。**

这套方案的优点：

- 总线扩展自然，后面再加节点只需要分配新的节点 ID。
- F103/F407 软件结构统一，联调时好比较。
- Linux 侧可以直接用 `candump` / `cansend` 做验证，不必先写额外程序。

## 总线与角色

### 物理层约束
- 总线波特率统一为 `500 kbps`。
- 所有节点必须共地。
- `TJA1050` 供电必须是 `5V`。
- 总线只保留两端 `120R` 终端电阻，不在每个分支板重复叠加终端。

### 角色分配
- `i.MX6ULL`：主站，负责发定向命令、接收两个节点的心跳和响应。
- `F103`：节点 1，节点 ID 设为 `0x01`。
- `F407`：节点 2，节点 ID 设为 `0x02`。

## 协议定义

### ID 规划
采用标准帧 `11-bit ID`，第一版固定如下：

| 用途 | 计算方式 | 示例 |
|------|---------|------|
| 心跳帧 | `0x100 + node_id` | `F103=0x101`，`F407=0x102` |
| 应答帧 | `0x200 + node_id` | `F103=0x201`，`F407=0x202` |
| 命令帧 | `0x300 + node_id` | `F103=0x301`，`F407=0x302` |

这样做的好处是：

- 主站抓包时一眼就能看出帧类型。
- 节点侧过滤和软件判断都很简单。
- 不需要扩展帧，也不需要复杂位域编码。

### 命令帧载荷
命令帧固定 `DLC=8`：

| Byte | 含义 |
|------|------|
| 0 | `cmd` 命令字 |
| 1 | `seq` 序号，主站递增 |
| 2 | `arg0` |
| 3 | `arg1` |
| 4 | `arg2` |
| 5 | `arg3` |
| 6 | 保留 |
| 7 | 保留 |

### 应答帧载荷
应答帧固定 `DLC=8`：

| Byte | 含义 |
|------|------|
| 0 | `cmd`，回显收到的命令字 |
| 1 | `seq`，回显收到的序号 |
| 2 | `status`，`0x00` 成功，非零为错误码 |
| 3 | `node_id` |
| 4 | `data0` |
| 5 | `data1` |
| 6 | `data2` |
| 7 | `data3` |

### 心跳帧载荷
心跳帧固定 `DLC=8`：

| Byte | 含义 |
|------|------|
| 0 | `node_id` |
| 1 | `heartbeat_counter` 低字节 |
| 2 | `heartbeat_counter` 高字节 |
| 3 | `last_cmd` |
| 4 | `last_status` |
| 5 | `tx_ok_counter` 低字节 |
| 6 | `rx_ok_counter` 低字节 |
| 7 | `error_flags` |

第一版只保留低字节计数，方便快速看趋势；后续如果业务需要更精细的统计，再扩成 16 位或 32 位计数。

## 第一版命令集

| 命令字 | 含义 | 说明 |
|------|------|------|
| `0x01` | `PING` | 节点收到后立即回包，证明定向收发链路正常 |
| `0x02` | `READ_STATUS` | 节点回传最近一次命令、心跳计数和错误状态 |
| `0x03` | `SET_HEARTBEAT_PERIOD` | 主站动态设置心跳周期，`arg0/arg1` 表示毫秒值 |

先把命令集压到最小范围，是为了让这版代码更像“总线节点模板”，而不是一开始就塞进业务逻辑。

## 节点运行行为

### F103/F407 共同行为
- 上电后初始化 UART、CAN、过滤器、中断和状态变量。
- 每 `1000 ms` 默认发一次心跳帧。
- 收到发给本节点 ID 的命令帧后：
  - 解析 `cmd/seq/arg`
  - 更新 `last_cmd`
  - 生成对应响应帧
  - 通过 UART 打印收包和发包结果
- 总线错误时，通过 UART 打印 `HAL_CAN_GetError()`、`ESR` 和节点状态。

### F407 特殊点
- 保留当前 `UART5` 作为日志输出。
- 不强行改系统时钟，先按当前 `HSI -> APB1=16 MHz` 重新计算 `500 kbps` 时序。
- 需要补齐 `CAN1_RX0_IRQHandler()` 和 `CAN1_SCE_IRQHandler()`。

### F103 特殊点
- 复用当前已验证的 `USART1` 日志框架。
- 把现有周期 `0x555` 演示发送改成“心跳 + 定向应答”模式。

## 联调流程
第一轮不编写 Linux 应用程序，直接用 `can-utils` 做主站验证：

1. `i.MX6ULL` 启 `can0`，波特率 `500000`。
2. 只接 F103，确认 `0x101/0x201` 正常。
3. 只接 F407，确认 `0x102/0x202` 正常。
4. F103、F407 同时挂上，确认：
   - `0x101` 和 `0x102` 都能稳定出现
   - 发 `0x301` 只由 F103 响应
   - 发 `0x302` 只由 F407 响应
   - 两者不会互相误应答

## 错误处理与调试策略
- 如果 STM32 串口持续打印发送超时，优先排查：
  - `TJA1050` 是否接 `5V`
  - CANH/CANL 是否接反
  - 是否共地
  - 是否只有两端终端电阻
  - Linux 侧 `can0` 是否真的 `up`
- 如果 Linux 侧 `cansend` 出现 `No buffer space available`：
  - 优先怀疑总线没有 ACK
  - 不要先怀疑应用层命令格式
- 如果只有一个节点正常，另一个节点完全无现象：
  - 先看该节点的 UART 启动日志
  - 再看 RX 中断是否进入
  - 最后再怀疑协议字段

## 本轮不做的事
- 不写 Linux 守护进程或 GUI。
- 不上扩展帧。
- 不做自动节点发现。
- 不把 F103/F407 抽成一个共享库工程。

当前最重要的是把“多节点最小闭环”做扎实，再考虑业务功能和代码抽象。
